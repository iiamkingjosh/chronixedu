1. Overview

Chronix Edu is fully deployed on Railway (API + Web) backed by Supabase (Postgres + Auth). The codebase is complete and self-contained. The items in this document are external service configurations — secrets, DNS records, and third-party account setup — that cannot be automated from within the codebase.

Work through the sections in order. Each section tells you exactly what to create, where to paste it, and what to verify.

Who should do this? A developer or DevOps person with admin access to: GitHub, Railway, Supabase, SendGrid, Sentry, and your DNS registrar.

2. Environment Variables Reference

These variables must be set in Railway (for both the API and Web services) and in GitHub Actions secrets for CI. Descriptions of each are below.

3. Sentry Error Tracking

What this does: Captures unhandled exceptions and API errors in real time. You get an alert the moment something breaks in production — before users report it.
1
Go to sentry.io and sign in (or create a free account).
2
Create a new project: Organization → Projects → Create Project.
Choose platform: Node.js / Express for the API. Name it chronixedu-api.
3
Copy the DSN shown after project creation (looks like https://abc123@o12345.ingest.sentry.io/789).
4
Repeat steps 2–3 for a second project: Next.js, name it chronixedu-web.
5
Add the DSNs as Railway environment variables: 
# API service
SENTRY_DSN=https://<api-dsn>@o<id>.ingest.sentry.io/<project>

# Web service
NEXT_PUBLIC_SENTRY_DSN=https://<web-dsn>@o<id>.ingest.sentry.io/<project>
6
Also add both DSNs to GitHub Actions secrets (see Section 5) so CI test runs can report errors too.
Verify: Deploy the app, then deliberately visit a non-existent API route. You should see an event appear in the Sentry dashboard within 30 seconds.

4. SendGrid Email Service

What this does: Sends welcome emails to new parent and student accounts with their login credentials and a temporary password.
1
Go to sendgrid.com and sign in (or create a free account — 100 emails/day free).
2
Verify your sender domain or email: Settings → Sender Authentication → Domain Authentication.
Recommended: authenticate your full domain (e.g., chronixedu.com) rather than a single email — this improves deliverability.
3
Create an API key: Settings → API Keys → Create API Key.
Permission: Restricted Access → Mail Send → Full Access. Copy the key immediately — it's shown only once.
4
Add to Railway environment variables (API service only): 
SENDGRID_API_KEY=SG.xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
SENDGRID_FROM_EMAIL=noreply@chronixedu.com
DNS records for domain authentication: SendGrid will give you CNAME records to add to your DNS. Add all of them — this is required for emails to land in the inbox and not spam.
Verify: Register a test student with a parent email you control. The parent should receive a welcome email with login credentials within 1–2 minutes.

5. GitHub Repository Secrets

These secrets are used by the GitHub Actions CI pipeline (.github/workflows/ci.yml) for running tests and triggering Railway deploys.

1
Go to your GitHub repository → Settings → Secrets and variables → Actions → New repository secret.
2
Add each of the following secrets:
Secret NameValue
DATABASE_URLYour Supabase PostgreSQL connection string (same as Railway)
JWT_SECRETSame JWT secret as Railway — must match exactly
SUPABASE_URLYour Supabase project URL
SUPABASE_PUBLISHABLE_KEYSupabase anon/publishable key
SUPABASE_SERVICE_ROLE_KEYSupabase service role key
SENTRY_DSNSentry API project DSN
NEXT_PUBLIC_SENTRY_DSNSentry Web project DSN
RAILWAY_DEPLOY_HOOKRailway deploy hook URL — see Section 6
Why duplicate with Railway? Railway variables are only available to running services. GitHub Actions needs its own copies to run tests in the CI environment. The values must be identical.

6. Railway Deployment

6.1 Get the Deploy Hook URL

1
In Railway: open your project → select your API or Web service → Settings → Deploy → Deploy Hook.
2
Click Generate webhook. Copy the URL (looks like https://backboard.railway.app/webhooks/deploy/xxx).
3
Paste it as the GitHub secret RAILWAY_DEPLOY_HOOK (Section 5). From now on, every push to main that passes CI will trigger a Railway redeploy automatically.
One hook per service: Railway generates one hook per service. If you have both an API and Web service, you may need to chain both hooks in the CI step, or use Railway's GitHub integration instead (which auto-deploys on push without needing a hook). Check your Railway plan for available options.

6.2 Set All Environment Variables in Railway

1
In Railway: select each service → Variables tab → add all variables from Section 2.
2
Set NODE_ENV=production on both services.
3
On the Web service, set NEXT_PUBLIC_API_URL to the Railway-assigned URL of your API service (or your custom domain if already set up).

6.3 Run Database Migrations

Do this before first launch. The migration runner applies all pending SQL migrations to your production database. Run it once after initial deploy, then it's automatic on subsequent deploys.
1
In Railway: open the API service → Settings → Deploy → Start Command.
Temporarily change to: npm --workspace=@chronixedu/api run migrate && npm --workspace=@chronixedu/api start
(Or run the migrate command once manually via Railway's shell, then revert.)
2
Alternatively, run migrations locally against the production DB: 
DATABASE_URL="your-production-db-url" npm run migrate
3
Verify: check Supabase Table Editor — you should see a schema_migrations table with all migration filenames listed.

7. Custom Domain & DNS

1
In Railway: select your Web service → Settings → Networking → Custom Domain → enter your domain (e.g., app.chronixedu.com).
2
Railway will show you a CNAME target (e.g., xyz.up.railway.app).
3
In your DNS registrar (e.g., Cloudflare, Namecheap), add: 
# Web frontend
Type:  CNAME
Name:  app
Value: <railway-cname-target>
TTL:   Auto

# API (if using custom domain for API too)
Type:  CNAME
Name:  api
Value: <railway-api-cname-target>
TTL:   Auto
4
Wait for DNS propagation (usually 5–30 minutes). Railway automatically provisions an SSL certificate via Let's Encrypt — no action needed.
5
Update your Railway Web service: set NEXT_PUBLIC_API_URL=https://api.chronixedu.com (your actual API domain).
Update your Railway API service: set CORS_ORIGIN=https://app.chronixedu.com.
Verify: Visit https://app.chronixedu.com — should load with a padlock. Visit https://api.chronixedu.com/health — should return JSON with "status": "ok".

8. Supabase Backups

What Supabase provides by default: Free plan: no automated backups. Pro plan ($25/mo): daily backups retained for 7 days. Enterprise: point-in-time recovery (PITR).

Option A — Upgrade to Supabase Pro (Recommended for production)

1
Supabase Dashboard → Settings → Billing → Upgrade to Pro.
2
Once on Pro: Settings → Database → Backups — daily backups appear here automatically. You can download or restore from this panel.

Option B — Manual pg_dump via Railway Cron (Free alternative)

1
Add a Railway Cron job service with schedule 0 2 * * * (2 AM daily).
2
Run script: 
pg_dump "$DATABASE_URL" | gzip > /backups/chronixedu-$(date +%Y%m%d).sql.gz
Mount a Railway volume to /backups for persistence.
Test your restore procedure before going live. A backup you've never restored is an untested backup.
To test: create a throwaway Supabase project, restore the dump, verify data integrity.

9. Uptime Monitoring

What this does: Pings your /health endpoint every 1–5 minutes and alerts you (email/SMS/Slack) if the API goes down. The /health endpoint is already built — it checks both the server and the database connection.

Using UptimeRobot (Free)

1
Go to uptimerobot.com → Create free account → Add New Monitor.
2
Settings: 
Monitor Type:  HTTP(s)
Friendly Name: Chronix Edu API
URL:           https://api.chronixedu.com/health
Monitoring Interval: 5 minutes
3
Under Alert Contacts: add your email and optionally a Slack webhook URL.
4
Add a second monitor for the frontend: 
URL: https://app.chronixedu.com
Expected status code: 200
Verify: UptimeRobot will send a test alert when you first save the monitor. Confirm you receive it.

10. Final Go-Live Checklist

Work through this in order before announcing the platform to users.

Infrastructure

  • ☐ Railway API service deployed and healthy (/health returns 200 ok)
  • ☐ Railway Web service deployed and loading
  • ☐ All environment variables set on both services
  • ☐ Database migrations run — schema_migrations table shows all entries
  • ☐ Custom domains configured, SSL certificates active (padlock visible)
  • ☐ CORS origin matches your production web domain exactly

External Services

  • ☐ Sentry projects created, DSNs configured, test error received
  • ☐ SendGrid API key added, sender domain verified, test welcome email received
  • ☐ Railway deploy hook added to GitHub secrets
  • ☐ GitHub Actions CI passing on main branch (check Actions tab)
  • ☐ UptimeRobot monitors active and alert email received
  • ☐ Supabase backups enabled (Pro plan) or manual cron configured

Functional Smoke Test

  • ☐ Super admin can log in
  • ☐ Super admin can create a school
  • ☐ Principal / Admin can log in to school dashboard
  • ☐ Registrar can add a student + parent — welcome email arrives
  • ☐ Parent can log in with temporary password
  • ☐ Teacher can take attendance and enter scores
  • ☐ Report card generation produces a PDF
  • ☐ Fee invoice creation and payment recording work
  • ☐ /health shows db: "ok" with a latency reading

Security

  • ☐ No .env files committed to git (git log --all -- '*.env' returns nothing)
  • ☐ chronixedu-test-credentials.html NOT committed to git
  • ☐ Supabase service role key is NOT exposed in any client-side code
  • ☐ Response headers include X-Frame-Options: SAMEORIGIN and Strict-Transport-Security
You're live! Once all boxes are checked, the platform is production-ready. Share the URL with your school administrators.
